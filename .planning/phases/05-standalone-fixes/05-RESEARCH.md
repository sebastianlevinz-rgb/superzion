# Phase 5: Standalone Fixes - Research

**Researched:** 2026-03-19
**Domain:** Phaser 3 game bug fixes (collision geometry, canvas drawing, UI text styling)
**Confidence:** HIGH

## Summary

Phase 5 addresses three independent shipping blockers in SuperZion, a Phaser 3.80.1 game with 100% procedural assets. Each fix is isolated to a single file or module with no shared system dependencies. The fixes span three distinct domains: physics/collision passability in a container-yard level, canvas 2D drawing geometry for an aircraft sprite, and text styling consistency across six gameplay scenes.

The codebase generates all textures via canvas API at runtime and uses Phaser's Arcade Physics for collisions. The three fixes can be planned and executed in any order since they touch completely separate files and systems.

**Primary recommendation:** Plan three independent tasks, one per requirement. Each is a surgical fix in a single file or tightly scoped module. The hardest is GAME-01 (container passability) which requires testing navigability after changes.

### Pattern 1: Procedural Texture Generation
**What:** All textures are generated at scene create() time using `document.createElement('canvas')` + Canvas 2D API, then registered with `scene.textures.addCanvas(key, canvas)`.
**When to use:** F-15 wing fix (GAME-02)
**Key detail:** The F-15 hangar texture is generated by `createF15Hangar()` in CinematicTextures.js. It creates a 600x400 canvas (modified) or 300x200 (original committed). The entire aircraft is drawn procedurally with `ctx.beginPath()` / `ctx.lineTo()` / `ctx.fill()`.

### Pattern 2: Physics Body Sizing
**What:** Phaser Arcade Physics uses `body.setSize(w, h)` to set collision rectangles independent of visual sprite size.
**When to use:** Container passability fix (GAME-01)
**Key detail:** Container sprites are 48x24 pixels but physics bodies are `setSize(40, 18)`. Player body is `setSize(14, 14)` with offset `setOffset(9, 13)`. Navigability depends on the gap between physics bodies, not visual sprites.

### Pattern 3: Controls Overlay Module
**What:** `showControlsOverlay(scene, text, opts)` creates a big centered overlay (3s) that transitions to a bottom bar.
**When to use:** UX-02 fix
**Key detail:** Big overlay uses 18px gold (#FFD700) text on black bg. Bottom bar uses 13px WHITE (#ffffff) text -- this is inconsistent with the requirement for bright yellow. Additionally, some scenes have their own `instrText` managed outside the overlay.

### Anti-Patterns to Avoid
- **Changing container physics body sizes without checking all interactions:** Guards (20x20 body) and workers also navigate the container yard. Widening corridors must maintain guard patrol paths.
- **Fixing only the big overlay or only the bottom bar:** Both overlay states need consistent yellow text on dark background per the requirement.

## Common Pitfalls

### Pitfall 1: Container Spacing Math
**What goes wrong:** Changing CONT_ROW_SPACING or CONT_COL_SPACING fixes one container grid but pushes containers outside the viewport or into walls/water.
**Why it happens:** Three container grids (north yard, south yard, east yard) have hardcoded base positions. Extra scattered containers in the east yard have hardcoded absolute coordinates.
**How to avoid:** Calculate total grid extent after spacing changes. North yard starts at y=130, south at y=340, east at y=160. Viewport is 960x540 with HUD_H=50. Top water wall ends at y=110. Verify no container exceeds these bounds.
**Warning signs:** Containers overlapping with water, walls, ships, cranes, or office buildings.

### Pitfall 2: Scene Key Collision
**What goes wrong:** Both `BeirutRadarScene.js` and `PortSwapScene.js` register with the key `'BeirutRadarScene'`. If both are loaded, only the last one registered takes effect.
**Why it happens:** PortSwapScene replaced BeirutRadarScene but reuses its scene key for backward compatibility with menu transitions.
**How to avoid:** Only modify `PortSwapScene.js` for GAME-01 fixes. The old `BeirutRadarScene.js` is dead code (not imported in main.js). Do NOT modify the old file.
**Warning signs:** Changes to the wrong BeirutRadarScene.js file.

### Pitfall 3: F-15 Wing Coordinate System
**What goes wrong:** Drawing wings with tip x-coordinate less than root x-coordinate produces forward-pointing wings.
**Why it happens:** In the side-view drawing, the nose is at the LEFT (low x) and the tail at the RIGHT (high x). For swept-back wings, the wing tip must have HIGHER x than the wing root.
**How to avoid:** Wing tip leading edge x must be GREATER than wing root leading edge x. In the committed code, tip is at x=100 and root at x=120 -- this is the bug. The tip needs to be at x >= 140.
**Warning signs:** Visually inspecting the generated texture to confirm wing direction.

### Pitfall 4: Controls Overlay -- Incomplete Coverage
**What goes wrong:** Fixing ControlsOverlay.js but missing the per-scene `instrText` that some scenes create independently.
**Why it happens:** Six scenes use `showControlsOverlay()` for the initial display, but four of those scenes also create their own `instrText` for dynamic status messages that change throughout gameplay. These independent text objects have different styling.
**How to avoid:** Audit each of the 6 gameplay scenes. The requirement is about the controls overlay text specifically, not all status text. Focus on the `showControlsOverlay` styling and the persistent bottom bar it creates.

### Pitfall 5: Uncommitted Local Changes
**What goes wrong:** CinematicTextures.js has extensive uncommitted modifications that include a rewritten F-15 with correct wings alongside completely unrelated sprite redesign work (VIS-01 scope, Phase 6).
**Why it happens:** The working copy has accumulated changes from multiple concerns.
**How to avoid:** The F-15 fix (GAME-02) should be scoped ONLY to the `createF15Hangar()` function. Either commit just that function's changes, or rewrite the wing geometry in the committed version's simpler format. Do NOT inadvertently commit the full VIS-01 sprite redesign as part of Phase 5.

## Critical Implementation Details

### GAME-01: Level 2 Container Passability

**Scene file:** `superzion/src/scenes/PortSwapScene.js`
**Scene key:** `'BeirutRadarScene'` (legacy name preserved)
**Menu entry:** Level 2: Operation Port Swap, accessed via `BeirutIntroCinematicScene`

**Container grid layout (3 yards):**
- North yard: 3 rows x 5 cols, base (160, 130), row_spacing=48, col_spacing=72
- South yard: 3 rows x 5 cols, base (160, 340), row_spacing=48, col_spacing=72
- East yard: 2 rows x 4 cols, base (560, 160), row_spacing=48, col_spacing=72
- East extras: 5 containers at absolute positions (560-704, 340-388)

**Physics dimensions:**
- Container body: 40w x 18h (visual: 48w x 24h)
- Player body: 14w x 14h
- Guard body: 20w x 20h
- Minimum corridor needed for player: 40 + 14 + clearance = ~58 horizontal, 18 + 14 + clearance = ~36 vertical

**Current clearance math:**
- Row spacing 48 between centers means 48 - 18 = 30px between body edges. Player needs 14px, so 30 - 14 = 16px clearance. This should be passable.
- Col spacing 72 between centers means 72 - 40 = 32px between body edges. Player needs 14px, so 32 - 14 = 18px clearance. This should be passable.

**Likely remaining issues to investigate:**
1. The target container selection logic -- does it always pick one the player can actually reach?
2. Do wall colliders (top water wall, boundary walls) block access to outer container rows?
3. Does the player spawn position at (80, 470) have a clear path to ALL container yards?
4. Are scattered east yard containers (lines 111-115) spaced too closely to each other? Their x-positions are 560, 632, 704 (72px apart = OK) and y-positions are 340, 388 (48px apart = OK). These should be fine.
5. Is the path between yards clear? The corridor between north and south yards is around y=280. Guard 2 patrols this corridor.

**Recommendation:** Play-test the level after making any changes. Focus on: (a) verifying the path from entrance (80, 490) to each container yard, (b) verifying the player can reach containers in ALL zones, (c) verifying the path from the target container back to the exit/entrance.

### GAME-02: F-15 Wing Direction

**File:** `superzion/src/utils/CinematicTextures.js`
**Function:** `createF15Hangar(scene)`
**Used by:** `DeepStrikeIntroCinematicScene` (Level 3 intro cinematic, Act 3)

**The bug (in committed code):** Wings point forward. The upper wing triangle is:
- Root: (120, 105) -- at fuselage center
- Tip: (100, 70) -- x=100 is TOWARD THE NOSE (x < root x)
- Trailing: (180, 100)

**The fix:** Swap wing tip to have x > root x. The tip should be at approximately x=160-180 to match F-15 swept-back geometry (~45-degree sweep).

**CRITICAL WARNING:** The working copy has extensive uncommitted changes to CinematicTextures.js that include:
1. Complete rewrite of `createSuperZionCinematic()` (128x192 detailed agent) -- this is VIS-01 / Phase 6 scope
2. Complete rewrite of `createF15Hangar()` (600x400 detailed F-15) -- includes the wing fix but also massive visual upgrade
3. Palette changes affecting multiple functions

**Recommended approach:** Fix ONLY the wing geometry in the committed version's simple format. Use `git stash` or work from the committed code, changing just the 4 wing vertex coordinates. This avoids pulling in Phase 6 scope.

### UX-02: Controls Overlay Readability

**File:** `superzion/src/ui/ControlsOverlay.js`
**Functions:** `showControlsOverlay()`, `addInstrTextBackground()`

**Current issues with the requirement "large bright yellow text on semi-transparent black background":**
1. Big overlay (first 3 seconds): ALREADY uses 18px #FFD700 (gold/yellow) on 0x000000/0.75 bg -- PASSES
2. Bottom bar: Uses 13px #ffffff (white) on 0x000000/0.7 bg -- FAILS (white not yellow, font too small)
3. Bottom bar text shadow is black blur, not yellow glow -- inconsistent with big overlay

**Fix scope:** Update `showControlsOverlay()` bottom bar text to use:
- Color: `#FFD700` (bright yellow, matching big overlay)
- Font size: `16px` or larger (currently 13px)
- Shadow: Yellow glow to match big overlay

**Note on `addInstrTextBackground()`:** This function is exported but not imported anywhere in the codebase (grep found no usage). It sets text to white (#ffffff) and 14px. If unused, it can be left as-is or updated for consistency, but it is not blocking.

**Note on per-scene instrText:** Several scenes (BomberScene, B2BomberScene, DroneScene, BossScene) create their own `this.instrText` for dynamic status messages. These are NOT the controls overlay -- they are gameplay status text that changes throughout the level. The UX-02 requirement specifically says "controls overlay," so these are out of scope for this fix.
