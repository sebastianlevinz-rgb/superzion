Modify the existing MusicManager class with three changes. Read the full file first.

**1. Fix stop() to use equal-power fade curve (replace lines 95-119):**
- Replace `linearRampToValueAtTime` with `setValueCurveAtTime` using a pre-computed equal-power Float32Array curve
- Pre-compute the fade-out curve once (lazy-init in stop or in _init): `this._fadeOutCurve = new Float32Array(64)` with `Math.cos((i / 63) * 0.5 * Math.PI)` values
- Scale the curve by `currentVol` before passing to `setValueCurveAtTime`
- Keep existing `_loopId` guard, cleanup scheduling, and `currentTrack = null` behavior
- Keep default `fadeTime = 0.5` but the method signature stays `stop(fadeTime = 0.5)` per user decision (callers already pass custom values like 0.3, 0.5, 1.0)
- Handle edge case: if `fadeTime <= 0` or `currentVol <= 0.001`, do immediate stop (cancelScheduledValues, set gain to 0, cleanup immediately, restore gain to 0.5)
- Always call `cancelScheduledValues(t)` before `setValueCurveAtTime` to avoid overlap DOMException (per research pitfall 1)

**2. Add crossfadeTo(trackName, startFn, duration = 1.5) method:**
- Default duration 1.5s per user decision (trance tracks need longer crossfade)
- If `this.currentTrack === trackName`, return immediately (same track, no-op -- per user decision)
- Call `this._init()` and guard `if (!this.ctx) return`
- Create a temporary GainNode (`inGain`) connected to `this.masterGain`, starting at gain 0
- Capture outgoing gain reference (`outGain = this.musicGain`) and its current volume
- Build two Float32Array(64) curves: outCurve uses `Math.cos(pct * 0.5 * Math.PI)` scaled by outVol, inCurve uses `Math.sin(pct * 0.5 * Math.PI)` scaled by target vol (0.5)
- Cancel scheduled values on both gains, then schedule both curves with `setValueCurveAtTime`
- Increment `_loopId`, clear `loopTimer`, capture old nodes as `const oldNodes = [...this._nodes]`, reset `this._nodes = []`
- Temporarily swap `this.musicGain = inGain`, call `startFn()` (which will use `_startLoop` connecting to the now-swapped `musicGain`), do NOT restore the old gain -- `musicGain` should remain as `inGain` going forward
- Note: `_startLoop` calls `stop(0.3)` internally. To avoid interference, the swap must happen BEFORE calling `startFn()`. The `stop(0.3)` inside `_startLoop` will fade the inGain (which just started at 0), which is harmless. But `_startLoop` also increments `_loopId` and sets up its own cleanup. This is actually correct: `_startLoop` manages its own lifecycle.
- Alternative cleaner approach: set `this.currentTrack = trackName` and `this._loopId++` before calling `startFn()`. Inside `startFn()`, `_startLoop` will call `stop(0.3)` on the inGain (which is at 0, so the fade is a no-op). Then `_startLoop` sets up the loop normally on inGain.
- Schedule cleanup of `oldNodes` after `(duration + 0.2) * 1000` ms, guarded by `_loopId`. In the cleanup: stop and disconnect each old node (try/catch), then `outGain.gain.cancelScheduledValues`, `outGain.gain.value = 0`, then `outGain.disconnect()`.

**3. Add shutdown() method:**
- Increment `_loopId`
- Clear `loopTimer` if exists
- If `ctx` and `musicGain` exist: `cancelScheduledValues`, set gain to 0 immediately
- Call `_cleanupNodes()` to force-drain all active nodes
- Set `currentTrack = null`
- This is for hard cleanup on scene destroy -- no fade, just immediate silence and cleanup

- Use ASCII art section dividers matching existing style: `// ═══════════...`
- Add new methods in the CORE section (after line 127, before the SYNTH PRIMITIVES section)
